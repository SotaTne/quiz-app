| id | question | answer | explanation |
|----|----------|--------|-------------|
| se07-001 | 設計工程で主に決めるのは「何を作るか」と「どう作るか」のどちらか。 | どう作るか | 要求・分析を実現する方法を決める。 |
| se07-002 | ソフトウェア全体の構成や分割を何というか。 | アーキテクチャ | 基本構造・枠組み。 |
| se07-003 | システムを複数の独立性のある単位に分けることを何というか。 | モジュール化 | 変更や理解を容易にする。 |
| se07-004 | 1つのモジュールが担当する責任のまとまりを何というか。 | 凝集度 | 高いほど望ましい。 |
| se07-005 | モジュール間の依存の強さを何というか。 | 結合度 | 低いほど望ましい。 |
| se07-006 | オブジェクトが外部へ提供する操作の境界を何というか。 | インターフェース | 利用側との契約になる。 |
| se07-007 | 設計の基本方針として、変更される部分を分離する考え方を何というか。 | 変更容易性 | 変更の影響範囲を小さくする。 |
| se07-008 | 要求を実際のプログラムに変換する工程は何か。 | 実装 | コーディングともいう。 |
| se07-009 | 画面などの表示を担当する部分を何と呼ぶか。 | View | MVCの構成要素。 |
| se07-010 | データや業務ルールを担当する部分を何と呼ぶか。 | Model | MVCの構成要素。 |
| se07-011 | 入力を受けてModelとViewをつなぐ部分を何と呼ぶか。 | Controller | MVCの構成要素。 |
| se07-012 | 高凝集・低結合を目指す主な理由は何か。 | 変更しやすくするため | モジュールの独立性を高める。 |
| se07-013 | Model・View・Controllerに分ける設計パターンは何か。 | MVC | 役割を分離する。 |
| se07-014 | 設計をプログラムに変換する作業は何か。 | コーディング | 実装の具体的作業。 |
| se07-015 | 外部へ提供する機能の取り決めを何というか。 | API | インターフェースの一種。 |
| se07-016 | 同じ処理を複数箇所に書かない原則は何か。 | DRY | Don't Repeat Yourself。 |
| se07-017 | 変更の影響を抑えるために分離する境界は何か。 | インターフェース | 実装と利用を分離する。 |
| se07-018 | 設計結果を実装前に確認する活動は何か。 | 設計レビュー | 欠陥を早期発見する。 |
