---
title: ソフトウェア工学 第01講 基礎
---

| id | question | answer | explanation |
|----|----------|--------|-------------|
| se01-001 | ソフトウェア工学の英語名は何か。 | Software Engineering | ソフトウェアの開発・運用・保守を工学的に扱う分野。 |
| se01-002 | ソフトウェア工学で重視される、開発への体系的・規律的・定量化可能なアプローチを何というか。 | systematic, disciplined, quantifiable approach | ISOの定義に含まれる考え方。 |
| se01-003 | プログラムだけでなく、仕様書や設計書などを含む成果物を何と呼ぶか。 | ソフトウェア | 製品として品質が求められる。 |
| se01-004 | ソフトウェア工学が対象とする、ソフトウェアの開発後の活動を2つ答えよ。 | 運用と保守 | 開発して終わりではない。 |
| se01-005 | ソフトウェア工学が誕生するきっかけとなった1968年の会議を何というか。 | NATOソフトウェア工学会議 | ドイツのガーミッシュで開催された。 |
| se01-006 | 遅れたプロジェクトに人員を追加すると、さらに遅れることがあるという法則は何か。 | ブルックスの法則 | 人月計算だけでは開発を扱えない。 |
| se01-007 | 順接・分岐・反復の制御構造を中心に記述する開発パラダイムは何か。 | 構造化 | 構造化プログラミングの基本。 |
| se01-008 | 変数と処理をまとめ、オブジェクト同士のやり取りで処理を記述する方法は何か。 | オブジェクト指向 | オブジェクトを中心に考える。 |
| se01-009 | システムを数学的に厳密な言語で記述・検証する手法は何か。 | 形式手法 | 要求を満たすか論理的に検証できる。 |
| se01-010 | エディタ、コンパイラ、リンカ、デバッガなどを統合した開発環境を何というか。 | IDE | Integrated Development Environmentの略。 |
| se01-011 | ソフトウェア開発において、万能な解決策は存在しないという考えを何というか。 | 銀の弾丸は存在しない | 本質的な難しさを一つの技術で消せない。 |
| se01-012 | ソフトウェア工学で、顧客の必要を引き出す活動は何か。 | 要求獲得 | ヒューマンスキルも重要。 |
| se01-013 | ソフトウェア開発の本質的な難しさを英語で何というか。 | essential difficulty | 技術だけでは完全に消せない。 |
| se01-014 | CASEは何の略か。 | Computer Aided Software Engineering | 開発を支援・自動化する。 |
| se01-015 | ソフトウェア工学で、開発後に必要な活動を2つ答えよ。 | 運用・保守 | リリース後も継続する。 |
| se01-016 | ソフトウェアの変更履歴を管理する代表的なOSSは何か。 | Git | 構成管理に利用する。 |
| se01-017 | 構造化の次に発展した代表的な開発パラダイムは何か。 | オブジェクト指向 | 変更に強い設計を目指す。 |
| se01-018 | ソフトウェアを製品として扱うとき特に求められるものは何か。 | 品質 | 工学的に品質を扱う。 |
